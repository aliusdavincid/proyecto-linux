# S1M01-0002 — La terminal: qué es un shell y bash

---

## Conceptos clave

| Término | Definición |
|---------|------------|
| Terminal | Interfaz que permite escribir comandos y recibir respuestas del sistema |
| Shell | Programa que interpreta los comandos que escribes en la terminal |
| Bash | El shell más común en Linux — Bourne Again Shell |
| Variable de entorno | Valor almacenado en memoria que el sistema usa para su configuración |
| $PATH | Variable que indica al sistema dónde buscar los programas ejecutables |
| /etc/shells | Archivo que lista todos los shells válidos instalados en el sistema |

---

## Lo que ejecuté

> Verificado en Debian GNU/Linux 13 (trixie) TTY y Linux Mint 22.3 (Zena).
> Las salidas son idénticas salvo donde se indica en la tabla de diferencias.

```bash
# echo $SHELL — muestra la ruta del shell activo en tu sesión
# lo importante: confirma que estás usando bash y dónde está instalado
aliusdavincid@debian:~$ echo $SHELL
/bin/bash

# echo $BASH_VERSION — muestra la versión exacta de bash en ejecución
# lo importante: útil para saber si ciertas funciones están disponibles
aliusdavincid@debian:~$ echo $BASH_VERSION
5.2.37(1)-release

# echo $0 — muestra el nombre del shell o script en ejecución actual
# lo importante: "bash" confirma que estás en una sesión interactiva
aliusdavincid@debian:~$ echo $0
bash

# cat /etc/shells — lista todos los shells válidos instalados en el sistema
# lo importante: solo los shells listados aquí pueden usarse como shell de login
aliusdavincid@debian:~$ cat /etc/shells
# /etc/shells: valid login shells
/bin/sh
/usr/bin/sh
/bin/bash
/usr/bin/bash
/bin/rbash
/usr/bin/rbash
/usr/bin/dash
/usr/bin/tmux

# bash --version — muestra información completa de la versión de bash
# lo importante: versión, arquitectura y licencia
aliusdavincid@debian:~$ bash --version
GNU bash, versión 5.2.37(1)-release (x86_64-pc-linux-gnu)
Copyright (C) 2022 Free Software Foundation, Inc.
Licencia GPLv3+: GPL de GNU versión 3 o posterior

# grep aliusdavincid /etc/passwd — muestra la entrada del usuario en el sistema
# lo importante: el último campo (/bin/bash) indica qué shell usa ese usuario por defecto
aliusdavincid@debian:~$ grep aliusdavincid /etc/passwd
aliusdavincid:x:1000:1000:nexus,,,:/home/aliusdavincid:/bin/bash

# echo $HOME — muestra el directorio personal del usuario activo
aliusdavincid@debian:~$ echo $HOME
/home/aliusdavincid

# echo $USER — muestra el nombre del usuario activo
aliusdavincid@debian:~$ echo $USER
aliusdavincid

# echo $PATH — muestra los directorios donde el sistema busca programas ejecutables
# lo importante: cuando escribes un comando el sistema lo busca en cada directorio en orden
aliusdavincid@debian:~$ echo $PATH
/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin
```

### Diferencias entre sistemas

| Comando | Debian | Mint |
|---------|--------|------|
| `echo $BASH_VERSION` | 5.2.37(1)-release | 5.2.21(1)-release |
| `echo $PATH` | Ruta corta y minimalista | Incluye `/snap/bin` y `/home/aliusdavincid/.dotnet/tools` |

## Lo que surgió mientras practicaba

**¿Por qué algunos comandos necesitan `echo` y otros no?**
Esta duda me surgió antes de ejecutar el primer comando. La respuesta es que `$SHELL`, `$PATH` y `$USER` no son comandos — son variables almacenadas en memoria. Si escribes solo `$SHELL` sin `echo`, bash intenta ejecutarlo como programa y falla. El `echo` le dice al sistema: "no lo ejecutes, solo muéstramelo". En cambio `bash --version` sí es un comando ejecutable y no necesita `echo`.

**¿Por qué las variables van en mayúsculas?**
Durante la sesión anoté esta duda directamente en la terminal. Es una convención del sistema: las variables de entorno del sistema siempre van en mayúsculas (`$SHELL`, `$PATH`, `$HOME`) para distinguirlas de las variables definidas por el usuario, que suelen ir en minúsculas. No es una regla obligatoria — es una convención que todos respetan para no confundirse.

**Descubrí tmux durante la práctica:**
Noté que tmux aparece listado en `/etc/shells` como si fuera un shell válido. Investigué y tmux es un multiplexor de terminal — permite tener varias ventanas abiertas dentro de una sola sesión TTY. Lo estoy usando en Debian y facilita el flujo de trabajo porque puedo tener la hoja de ruta en una ventana y los comandos en otra. Es una herramienta que encontré en el camino y que ya se volvió parte de mi forma de trabajar.

**El problema del teclado en inglés:**
En la TTY de Debian el layout es inglés y yo estoy acostumbrado al latinoamericano. Símbolos como `$`, `--` y `/` no están donde los espero. Ejecutar `loadkeys la-latin1` cambia el layout para esa sesión, pero la mejor solución a largo plazo es aprender a moverme con soltura en ambos — en entornos reales no siempre puedes elegir.

**Cómo capturar y transferir sesiones entre sistemas:**
Como trabajo en dual boot no puedo transferir archivos directamente. El flujo que desarrollé fue: activar `script` en Debian, montar el disco de Mint desde Debian y copiar el archivo a la carpeta de trabajo. Desde Mint abro el archivo y documento el apunte. Es un flujo manual pero funciona y lo aprendí practicando.

**Dos archivos `script` en lugar de uno:**
Decidí hacer un archivo de sesión separado para Debian y otro para Mint. Es más fácil leer y comparar dos archivos limpios que un solo archivo mezclado. El operador `>>` permite añadir al mismo archivo, pero dos archivos separados dan más claridad al momento de documentar.

## Lo que me pasó

**Error: `bash //version` en lugar de `bash --version`:**
```
bash: //version: No existe el fichero o el directorio
```
Bash interpretó `//version` como una ruta de directorio. El símbolo correcto es `--` (dos guiones), no `//`.

**Error: `cho $SHELL` en lugar de `echo $SHELL` en Mint:**
```
Orden «cho» no encontrada.
```
Mint fue más descriptiva en el error — sugirió paquetes similares incluyendo `echo`. Debian solo dice `orden no encontrada`. Esta diferencia muestra que Mint está más orientada al usuario que Debian.

**Error silencioso: `echo %PATH` en lugar de `echo $PATH`:**
```
%PATH
```
El sistema no dio error — simplemente imprimió `%PATH` como texto literal porque `%` no es el símbolo de variable en bash. El símbolo correcto es `$`. Este tipo de error es el más peligroso porque pasa desapercibido.

## Para qué sirve esto en la práctica

| Comando | Situación real |
|---------|----------------|
| `echo $SHELL` | Confirmar qué shell estás usando antes de escribir un script |
| `echo $BASH_VERSION` | Verificar compatibilidad antes de ejecutar un script en otro servidor |
| `cat /etc/shells` | Saber qué shells están disponibles antes de cambiar el shell de un usuario |
| `grep usuario /etc/passwd` | Verificar qué shell tiene asignado un usuario sin abrir archivos de configuración |
| `echo $PATH` | Diagnosticar por qué el sistema no encuentra un comando que instalaste |
| `echo $HOME` | Referenciar el directorio del usuario en scripts sin escribir la ruta completa |

## Lo que no haría / Lo que sí usaría

- ✅ Usaría `echo $PATH` como primer diagnóstico cuando un comando instalado no es encontrado
- ✅ Usaría `script` para grabar todas mis sesiones de práctica — es la forma más honesta de documentar lo que realmente pasó
- ❌ No cambiaría de bash a otro shell mientras estoy aprendiendo — la lógica es la misma en todos pero los comandos específicos cambian. Primero dominar bash
- ❌ No usaría `%` para referenciar variables — en bash el símbolo siempre es `$`

---

## Lo que me quedó claro

El shell es el intermediario entre lo que escribo y lo que hace el sistema. Las variables de entorno no son comandos — son valores en memoria que necesitan `echo` para poder verlas. En entornos reales casi siempre se trabaja con configuraciones mínimas, así que aprender bash bien antes de explorar otros shells es la decisión más inteligente. Cada error de tipeo en esta sesión me enseñó algo que no hubiera aprendido leyendo un manual.
