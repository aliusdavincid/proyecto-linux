# S1M01-0003 — El árbol de directorios: /, /etc, /var, /usr, /home, /proc

---

## Conceptos clave

| Término | Definición |
|---------|------------|
| Directorio raíz `/` | El nivel más alto del sistema de archivos — todo parte desde aquí |
| `/etc` | Archivos de configuración del sistema y los programas instalados |
| `/var` | Datos variables — logs, correos, bases de datos, cachés |
| `/usr` | Programas y recursos del sistema para todos los usuarios |
| `/home` | Directorios personales de cada usuario del sistema |
| `/proc` | Sistema de archivos virtual — muestra información del kernel y procesos en tiempo real |
| `cd ..` | Sube un nivel en el árbol de directorios |
| `cd /` | Va directamente al directorio raíz |
| `cd ~` | Va directamente al directorio del usuario activo |
| `pwd` | Muestra la ruta exacta donde estás en este momento |

---

## Lo que ejecuté

> Verificado en Debian GNU/Linux 13 (trixie) TTY y Linux Mint 22.3 (Zena).
> Las salidas son idénticas salvo donde se indica en la tabla de diferencias.

```bash
# cd .. — sube un nivel en el árbol de directorios
# lo importante: hay que usarlo dos veces desde ~ para llegar a /
aliusdavincid@debian:~$ cd ..
aliusdavincid@debian:/home$ cd ..
aliusdavincid@debian:/$

# cd / — va directamente al directorio raíz en un solo comando
# lo importante: es más directo que usar cd .. varias veces
aliusdavincid@debian:~$ cd /
aliusdavincid@debian:/$

# cd ~ — va directamente al directorio del usuario activo
aliusdavincid@debian:/$ cd ~
aliusdavincid@debian:~$

# cd ../.. — sube dos niveles en un solo comando
# lo importante: se pueden encadenar varios .. para subir varios niveles a la vez
aliusdavincid@debian:~$ cd ../..
aliusdavincid@debian:/$

# pwd — muestra la ruta exacta donde estás en este momento
# lo importante: úsalo para orientarte antes de ejecutar cualquier comando
aliusdavincid@debian:/$ pwd
/

# ls / — lista el contenido del directorio raíz
# lo importante: estos son todos los directorios base del sistema
aliusdavincid@debian:/$ ls /
bin  boot  dev  etc  home  initrd.img  initrd.img.old  lib  lib64
lost+found  media  mnt  opt  proc  root  run  sbin  srv  sys  tmp
usr  var  vmlinuz  vmlinuz.old

# ls /etc — muestra los archivos de configuración del sistema
# lo importante: aquí viven las configuraciones de casi todo el sistema
aliusdavincid@debian:/$ ls /etc

# ls /var — muestra los datos variables del sistema
# lo importante: aquí están los logs — si algo falla, empiezas a buscar aquí
aliusdavincid@debian:/$ ls /var
backups  cache  lib  local  lock  log  mail  opt  run  spool  tmp

# ls /usr — muestra los programas instalados del sistema
# lo importante: /usr/bin tiene los ejecutables disponibles para todos los usuarios
aliusdavincid@debian:/$ ls /usr
bin  games  include  lib  lib64  libexec  local  sbin  share  src

# ls /home — muestra los directorios de los usuarios del sistema
# lo importante: cada usuario tiene su carpeta personal aquí
aliusdavincid@debian:/$ ls /home
aliusdavincid

# ls /proc — muestra información del kernel y procesos activos en tiempo real
# lo importante: no es un directorio real — sus archivos no ocupan espacio en disco
aliusdavincid@debian:/$ ls /proc
1  14  155  1811  2084  35  58  72  88  bus  meminfo  cmdline
cpuinfo  meminfo  modules  mounts  version  uptime ...

# cat /proc/version — muestra la versión exacta del kernel en ejecución
aliusdavincid@debian:/$ cat /proc/version
Linux version 6.12.74+deb13+1-amd64 ...

# cat /proc/uptime — muestra cuántos segundos lleva el sistema encendido
# lo importante: el primer número son segundos activo, el segundo segundos idle
aliusdavincid@debian:/$ cat /proc/uptime
325.99 2444.36

# cat /proc/meminfo — muestra el estado actual de la memoria RAM
# lo importante: MemTotal y MemAvailable muestran cuánta RAM tienes y cuánta está libre
aliusdavincid@debian:/$ cat /proc/meminfo
MemTotal:       16210doorbell kB
MemFree:         XXXX kB
MemAvailable:    XXXX kB
...

# cd /root — intento de entrar al directorio del superusuario sin permisos
# lo importante: el sistema lo bloquea directamente
aliusdavincid@debian:/$ cd root
bash: cd: root: Permiso denegado

# su — cambia al usuario root con su contraseña
# lo importante: en Debian funciona si conoces la clave de root
aliusdavincid@debian:/$ su
Contraseña:
root@debian:/# cd root
root@debian:~# exit

# whoami — confirma con qué usuario estás operando en este momento
aliusdavincid@debian:/$ whoami
aliusdavincid
```

### Diferencias entre sistemas

| Situación | Debian | Mint |
|-----------|--------|------|
| Escalar a root | `su` funciona con clave de root | `su` falla — root no tiene clave activa en Mint |
| Acceso root alternativo | `su` directo | `sudo -s` o `sudo -i` |
| Directorio raíz `ls /` | Sin `cdrom`, sin `swapfile` visible | Incluye `cdrom` y `swapfile` |
| `sudo -i` vs `sudo -s` | Ambos funcionan igual | Se comportan diferente — `-i` simula login completo, `-s` solo abre shell |

## Lo que surgió mientras practicaba

**Descubrí dos formas de llegar al directorio raíz:**
Al inicio usé `cd ..` dos veces para subir desde mi carpeta de usuario hasta `/`. Luego me pregunté si había una forma más directa y probé `cd /` — llega en un solo comando sin importar dónde estés. También descubrí que `cd ../..` encadena dos niveles en un solo comando. La forma más directa siempre es `cd /`.

**`pwd` como brújula:**
Mientras navegaba entre directorios perdí la noción de dónde estaba. Ahí entendí para qué sirve `pwd` — no es un comando que ejecutas de forma rutinaria sino una brújula que usas cuando te desorientas. Desde entonces lo ejecuto cada vez que no estoy seguro de mi ubicación antes de hacer algo.

**`/proc` no ocupa espacio real en disco:**
Al listar `/proc` aparecen decenas de carpetas numeradas que corresponden a los procesos activos. Pero si intentas medir su tamaño no ocupa nada — porque no existe en el disco. Es un sistema de archivos virtual que el kernel genera en memoria al momento de arrancar. Cada vez que el sistema se reinicia, `/proc` se recrea desde cero. Eso explica por qué sus archivos desaparecen al apagar — nunca estuvieron en el disco.

**El intento de entrar a `/root` y lo que descubrí sobre `su` vs `sudo`:**
Intenté entrar a `/root` directamente y recibí `Permiso denegado`. Luego probé `sudo cd root` y el sistema me respondió algo inesperado:
```
sudo: cd: command not found
sudo: "cd" is a shell built-in command, it cannot be run directly.
```
`cd` no es un programa externo — es un comando interno del shell y `sudo` no puede ejecutarlo directamente. La solución correcta es escalar a root primero con `su` y luego navegar. En Mint el comportamiento fue diferente — `su` falla porque root no tiene clave activa por defecto, y la forma correcta es `sudo -s` o `sudo -i`. Son dos filosofías distintas sobre cómo gestionar el acceso root.

**`sudo -i` vs `sudo -s` me confundieron:**
Probé ambos en Mint y se comportan diferente. `sudo -s` abre un shell como root pero mantiene el entorno del usuario actual. `sudo -i` simula un login completo como root — carga el entorno de root desde cero. En Debian ambos se comportan de forma más similar. Este es el tipo de diferencia que solo descubres ejecutando, no leyendo.

**Reflexión sobre el usuario root:**
Tener la clave de root en un servidor de producción es una responsabilidad enorme. Si una empresa depende de Linux para operar y alguien con acceso root comete un error, puede romper el sistema completo. Tiene sentido que Mint deshabilite root por defecto y fuerce el uso de `sudo` — es más seguro que dar acceso directo al nivel más alto del sistema.

**`/etc` tiene muchísimos archivos:**
Al listar `/etc` en Mint aparecieron cientos de archivos. Cada uno es la configuración de algo — apache, ssh, cron, usuarios, red. Por ahora no toco nada ahí, pero es el directorio que más voy a usar cuando empiece a administrar servicios. Es el corazón de la configuración del sistema.

## Lo que me pasó

**`sudo cd root` no funciona:**
```
sudo: cd: command not found
sudo: "cd" is a shell built-in command, it cannot be run directly.
```
`cd` es un comando interno del shell, no un programa externo. `sudo` solo puede ejecutar programas externos. Para navegar como root hay que escalar primero con `su` o `sudo -s` y luego usar `cd`.

**`su` falla en Mint sin contraseña de root:**
```
su: Fallo de autenticación
```
En Mint el usuario root existe pero no tiene contraseña activa por defecto. La forma correcta de obtener acceso root en Mint es `sudo -s` o `sudo -i`, no `su` directo.

**Error de sintaxis al escribir paréntesis en `echo`:**
```
bash: error sintáctico cerca del elemento inesperado `('
```
Intenté escribir una anotación con paréntesis dentro de un `echo` sin comillas. Bash interpretó el paréntesis como parte de la sintaxis del shell. La solución es siempre envolver el texto de `echo` entre comillas dobles.

## Para qué sirve esto en la práctica

| Comando | Situación real |
|---------|----------------|
| `cd /` | Orientarte en un servidor desconocido — empezar desde la raíz |
| `pwd` | Confirmar dónde estás antes de ejecutar un comando destructivo |
| `ls /etc` | Buscar el archivo de configuración de un servicio que está fallando |
| `ls /var/log` | Primera parada cuando algo falla — los logs están aquí |
| `cat /proc/meminfo` | Diagnosticar problemas de rendimiento — ver cuánta RAM está disponible |
| `cat /proc/version` | Confirmar versión del kernel en un servidor remoto |
| `cat /proc/uptime` | Verificar cuánto tiempo lleva corriendo el sistema sin reiniciar |
| `sudo -i` | Obtener acceso root completo en sistemas basados en Ubuntu/Mint |
| `su` | Obtener acceso root en Debian cuando conoces la clave de root |

---

## Lo que no haría / Lo que sí usaría

- ✅ Usaría `cd /` siempre que necesite orientarme — es el punto de partida más claro
- ✅ Usaría `pwd` antes de cualquier operación que no pueda deshacerse
- ✅ Iría primero a `/var/log` si algo falla en el sistema — los logs son la primera pista
- ❌ No tocaría nada en `/etc` sin entender exactamente qué archivo estoy modificando — una mala configuración puede romper un servicio entero
- ❌ No usaría `sudo cd` — no funciona porque `cd` es un built-in del shell
- ❌ No trabajaría de forma permanente como root — escalar solo cuando es necesario y volver al usuario normal inmediatamente

---

## Lo que me quedó claro

El árbol de directorios de Linux no es arbitrario — cada directorio tiene una función específica y todos dependen del directorio raíz `/`. Lo más revelador de esta sesión fue `/proc`: un directorio que parece real pero no existe en el disco — el kernel lo genera en memoria al arrancar y desaparece al apagar. Eso me hizo entender que en Linux no todo lo que ves es un archivo real. Y sobre root: tener ese acceso es poder total sobre el sistema, pero también responsabilidad total. Tiene sentido aprenderlo con cuidado.
